## 🎯 Introduction

This **2-hour** hands-on workshop will guide you through building a **Banking System for Contoso Bank**, a fictitious financial institution. You'll use **GitHub Copilot** as your development assistant to build a complete application with C# and .NET 8. You'll learn to:

- ✅ Use Copilot's autocomplete and inline suggestions
- ✅ Write effective prompts that guide Copilot with intent
- ✅ Build a REST API with Minimal APIs and automatic Swagger documentation
- ✅ Develop a frontend that consumes the API
- ✅ Generate AI-assisted unit tests
- ✅ Use Copilot Chat to explain, fix, and refactor code

### Project Standards

| Aspect | Standard |
|--------|----------|
| Type | REST API |
| Technology | .NET 8 (LTS) |
| API Style | Minimal APIs |
| API Documentation | Swagger UI (Swashbuckle) |
| Language | English (code, comments, documentation) |
| Database | In-memory (C# lists and dictionaries) |
| Frontend | HTML + vanilla JavaScript + Bootstrap 5 (no frameworks) |
| Testing | xUnit + WebApplicationFactory |

### Scenario: Contoso Bank 🏦

**Contoso Bank** is a financial institution that needs a digital system to manage:

- **Customers**: personal and contact information
- **Bank accounts**: account types, balances, statuses
- **Transactions** *(bonus)*: deposits, withdrawals, transfers

> 💡 Complexity is **intentionally low**. The goal is NOT to build a robust app, but to show how **Copilot accelerates every phase of development**.

> 📝 **About transactions:** The guided exercises cover **Customers** and **Accounts** as core functionality. **Transactions** are presented as an optional challenge at the end of Exercise 1 for those who finish early. Don't worry if you don't complete them — the workshop's value is in the process, not in finishing everything.

4. C# services process the logic and store data in in-memory collections
5. Swagger UI is available at `/swagger` to explore and test the API

## 🧠 GitHub Copilot Key Concepts

### What is GitHub Copilot?

GitHub Copilot is an **AI-powered programming assistant** that integrates directly into your code editor. It works as a **pair programmer** that:

- Suggests complete lines or blocks of code as you type
- Understands your project's context (file names, comments, existing code)
- Learns from your patterns and adapts to your style

### The Art of Prompting in Copilot

The key to getting good results from Copilot is **how you describe what you need**. Descriptive, intentional comments guide Copilot better than rigid step-by-step instructions.

**❌ Weak prompt:**
```
Build my banking app
```

> What's wrong? No business context, no technology, no expected output. Copilot has to **guess** everything.

**✅ Effective prompt:**
```csharp
// Minimal API endpoint to get the current balance of a Contoso Bank account
// Takes the account number as a route parameter
// Returns the available balance and last update date as JSON
// Includes OpenAPI documentation with .WithTags() and .Produces<>()
```

> Notice how the second comment gives Copilot **context** (Contoso Bank), **intent** (get balance), **details** (parameters and return value), and **technology** (Minimal APIs with OpenAPI). The more specific your intent, the better the suggestions.

> 📚 **Want more best practice examples?** Check out the [github/awesome-copilot](https://github.com/github/awesome-copilot) repository — it contains community-contributed instructions, agents, and configurations to get the most out of GitHub Copilot.

### GitHub Copilot Chat Modes

> ⚠️ **Important note:** The mode interface (icons, selector location, names) may vary depending on your VS Code version and the GitHub Copilot extension. If you see a different interface than described here, check with your instructor or review the [official documentation](https://docs.github.com/en/copilot).

GitHub Copilot has three main operating modes:

#### 1️⃣ Ask Mode 💬

| Aspect | Detail |
|--------|--------|
| Icon | 💬 Chat bubble |
| Function | Only answers questions, does **NOT** modify files |
| Best for | Exploring, understanding, planning, learning |

**Example:**
```
[Ask Mode]
"What's the best way to implement a REST API with Minimal APIs and Swagger?"

→ Copilot EXPLAINS the options but does NOT create files
```

#### 2️⃣ Agent Mode 🤖

| Aspect | Detail |
|--------|--------|
| Icon | 🤖 Robot or sparkle |
| Function | **CAN** create and modify files automatically |
| Best for | Implementing changes, creating code, refactoring |

**Example:**
```
[Agent Mode]
"Create a REST API with Minimal APIs to manage Contoso Bank customers"

→ Copilot CREATES the files with all the code
```

#### 3️⃣ Plan Mode 📋

| Aspect | Detail |
|--------|--------|
| Icon | 📋 List or document |
| Function | Generates a detailed plan **BEFORE** executing |
| Best for | Complex tasks involving multiple files |

**Example:**
```
[Plan Mode]
"Implement complete transaction banking functionality 
with model, service, and endpoints"

→ Copilot SHOWS the plan:
  1. Create Transaction.cs (model)
  2. Create TransactionService.cs (business logic)
  3. Add transaction endpoints in Program.cs
  4. Register the service in the DI container

→ You APPROVE each step before it executes
```

#### Mode Comparison

| Feature | Ask 💬 | Agent 🤖 | Plan 📋 |
|---------|--------|----------|---------|
| Modifies files | ❌ No | ✅ Yes | ✅ Yes (with approval) |
| Speed | Fast | Fast | Slower |
| Control | N/A | Low | High |
| Best for | Learning | Implementing | Complex tasks |
| Risk | None | Medium | Low |

### Copilot Customization: Instructions, Prompt Files, Agents, and Skills

GitHub Copilot can be customized at several levels using configuration files within your repository. These are the main components of the customization system:

| Component | Location | Purpose | When to use |
|-----------|----------|---------|-------------|
| **Instructions** | `.github/copilot-instructions.md` | Global rules Copilot applies **always** across the project (conventions, stack, patterns) | From the start of the project — defined once |
| **Prompt Files** | `.github/prompts/*.prompt.md` | Reusable prompts for repetitive tasks, invoked manually with `/name` | When you repeat the same type of request frequently |
| **Custom Agents** | `.github/agents/*.md` | Specialized versions of Copilot with a defined role, tools, and behavior | When you need a "specialist" (frontend, testing, docs, security) |
| **Skills** | `.github/skills/*/SKILL.md` | Instruction packages with bundled scripts and resources for multi-step tasks | When the task requires auxiliary files or bundled scripts |

Think of **instructions** as team rules everyone follows, **prompt files** as reusable templates for common tasks, **agents** as specialist teammates you can invoke for specific tasks, and **skills** as mini-runbooks that include everything needed to execute a complex task.

> 📚 In this workshop we'll use **instructions** (Step 1.3) and create a **custom agent** (Step 2.2). To explore more components, check the [github/awesome-copilot](https://github.com/github/awesome-copilot) repository.

### Special Commands (/command)

| Command | Description | Usage example |
|---------|-------------|---------------|
| `/tests` | Generate unit tests | Select code → `/tests` |
| `/doc` | Generate documentation | Select method → `/doc` |
| `/fix` | Propose error corrections | Select buggy code → `/fix` |
| `/explain` | Explain selected code | Select code → `/explain` |

### Step 1.1: Explore with Ask Mode 🔍

> 💡 **IMPORTANT:** Make sure you're in **Ask Mode** (chat bubble icon 💬). This mode does NOT modify files, it only answers questions.

📍 **How to activate Ask Mode:**
1. Open Copilot Chat (`Ctrl+Shift+I` / `Cmd+Shift+I`)
2. Look for the mode selector at the top

🤖 **PROMPT — Copy and paste into Copilot Chat:**

```

```

📝 **Observe:** Copilot responds with detailed information but does **NOT create any files**. This is ideal for the exploration and planning phase.

> 🌟 **Wow moment:** Notice how Copilot understands the banking domain and suggests a coherent architecture without you giving excessive technical details.

### Step 1.2: Document the System Specification 📄

> 💡 **Why this step?** Before writing code, it's good practice to capture the specification of what you're going to build in a document within the repository. This document serves two purposes: it acts as a reference for you and your team, and it's the **foundation for Copilot's agents, skills, and prompt files to have detailed context** about what they need to build. In agentic development, the quality of the specification determines the quality of the output.

> 💡 **IMPORTANT:** Switch to **Agent Mode** (robot icon 🤖). This mode **CAN** create and modify files.

📍 **How to activate Agent Mode:**
1. In Copilot Chat, look for the mode selector
2. Select **"Agent"** or the robot/sparkle icon

🤖 **PROMPT in Agent Mode:**

```
Based on our previous conversation about the banking system design, create the file docs/spec.md with the technical specification for the Contoso Bank project.

The document should include:

1. Executive summary of the system
2. Domain entities with their fields and data types
3. List of REST endpoints (HTTP method, route, description, response codes)
4. Business rules (balance validations, account types, statuses)
5. Technology stack (.NET 8, Minimal APIs, Swashbuckle, xUnit)
6. Expected project folder structure
7. Sample data for each entity (3 customers, 3 accounts)

Format: Structured Markdown with tables where applicable.
```

📝 **Why is this important?** This `docs/spec.md` file becomes the project's **contract**. When you later use Copilot in Agent mode to generate code, models, or tests, you can reference it in your prompts: *"Implement the customer endpoints according to the specification in docs/spec.md"*. This is much more precise than repeating the requirements every time.

> 🌟 **Wow moment:** Copilot takes the Step 1.1 conversation as context and generates a complete, coherent specification document. This **explore → document → implement** pattern is the foundation of agentic development: you define what you want with precision, then let Copilot execute against that definition.

### Step 1.3: Create Copilot Instructions for the Project

> 💡 **Why now?** The `copilot-instructions.md` file configures Copilot to follow the project's standards in **all** files it generates from this point forward. Creating it before writing code ensures that models, the API, and the frontend are generated with the correct conventions from the start. Unlike the specification (`docs/spec.md`) which describes **what** to build, the instructions define **how** Copilot should behave.

🤖 **PROMPT in Agent Mode:**

```
Create the file .github/copilot-instructions.md with instructions for Copilot to act as a C# and .NET expert for Contoso Bank:

# GitHub Copilot Instructions - Contoso Bank Project

## Code Standards
| Aspect | Standard |
|--------|----------|
| Technology | .NET 8 (LTS) |
| API Style | Minimal APIs |
| Swagger | Swashbuckle |
| Style | Microsoft C# conventions |
| Documentation | XML comments (///) |

## Naming Conventions
- Public properties and methods: PascalCase (GetCustomers, CreateAccount)
- Local variables and parameters: camelCase (customerId, currentBalance)
- Classes and records: PascalCase (Customer, BankAccount, CustomerService)
- Constants: PascalCase (MinimumBalance, MaxTransferAmount)
- Interfaces: IPrefix + PascalCase (ICustomerService)

## Project Context
This is a banking system for Contoso Bank that manages customers, accounts, and transactions. It uses in-memory data (C# collections) with no external database. Uses Minimal APIs (not Controllers). Uses records for DTOs when appropriate. The full specification is in docs/spec.md.
```

### Step 1.7: Create the Application with Minimal APIs and Swagger

Now we'll ask Copilot to generate the main application. Notice how with a concise, intent-oriented prompt, Copilot can generate a complete app.

🤖 **PROMPT in Agent Mode:**

```
Update ContosoBank/Program.cs as the main application for Contoso Bank.

Configure:
1. Swashbuckle for Swagger UI with title "Contoso Bank API" and version "v1"
2. Register CustomerService and AccountService as Singleton in the DI container
3. Enable static files (UseStaticFiles) to serve the frontend from wwwroot/
4. Redirect "/" to "/index.html" with a simple MapGet
5. Minimal API endpoints with groups:
   - var customers = app.MapGroup("/api/customers").WithTags("Customers")
     GET /           → get all
     GET /{id}       → get by id (returns 404 if not found)
     POST /          → create (returns 201 with Location header)
     PUT /{id}       → update (returns 404 if not found)
     DELETE /{id}    → delete (returns 404 if not found)
   - var accounts = app.MapGroup("/api/accounts").WithTags("Accounts")
     Same CRUD endpoints
6. Use .WithName(), .WithOpenApi() and .Produces<T>() to document each endpoint
7. Add app.UseSwagger() and app.UseSwaggerUI()
```

> 💡 **If the suggestion is missing something** (e.g., missing Swagger documentation or the frontend route), try a follow-up prompt like: *"Add .WithOpenApi() and .Produces<>() to all endpoints to improve Swagger documentation"* or *"Add UseStaticFiles and a redirect from / to /index.html"*. Iterating is a natural part of working with Copilot.

## 🙋 FAQ

### Why do we use in-memory data instead of Entity Framework?

| Advantage | Detail |
|-----------|--------|
| No installation | No SQL Server, SQLite, or any DB engine required |
| Fast | Instant operations, ideal for a workshop |
| Portable | Works the same on any machine with .NET 8 |
| Simple | Lets us focus on Copilot, not DB configuration |

**How to migrate to Entity Framework Core?**
1. Add the `Microsoft.EntityFrameworkCore` and `Microsoft.EntityFrameworkCore.Sqlite` packages
2. Create a `DbContext` with `DbSet<Customer>`, `DbSet<Account>`, etc.
3. Replace in-memory services with EF Core repositories
4. Copilot can help: *"Migrate the in-memory services to Entity Framework Core with SQLite according to the specification in docs/spec.md"*

### Why .NET 8 and not .NET 9?

.NET 8 is the **LTS** (Long Term Support) version with support until November 2026. Additionally, .NET 9 removed Swashbuckle from the default template in favor of `Microsoft.AspNetCore.OpenApi`, which adds an unnecessary extra configuration step for a workshop. If your organization already uses .NET 9, the migration is minimal: just add Swashbuckle manually or use the new native OpenAPI package.

### Why Minimal APIs and not Controllers?

| Minimal APIs | Controllers |
|--------------|-------------|
| Less ceremony, more concise code | More structure, more verbose |
| Everything in one file to start | Requires Controllers/ folder + separate files |
| Ideal for small/medium APIs | Ideal for large APIs with many conventions |
| Recommended pattern since .NET 7+ | Classic pattern, widely known |

For a 2-hour workshop, Minimal APIs lets us focus on **Copilot** instead of Controller ceremony, `ControllerBase` inheritance, and attributes.

### Does Copilot generate different code for each person?

Yes, and that's **intentional**. Copilot learns from context (your code, your comments, your style) and may generate slightly different solutions for each person. This is a feature, not a bug.

### What if Copilot suggests incorrect code?

1. **Don't blindly accept** — always review suggestions
2. Press `Esc` to reject and adjust your comment/prompt
3. Use `Alt+]` to see alternative suggestions
4. Remember: Copilot is an **assistant**, not a developer replacement

### What if my Copilot interface looks different?

The GitHub Copilot interface (modes, icons, selectors) is updated frequently. If the icons or mode locations don't exactly match what's described in this workshop, check the [official VS Code + Copilot documentation](https://code.visualstudio.com/docs/copilot/overview) or ask the instructor.
